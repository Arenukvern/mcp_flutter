# ADR 0007 — Web headful tab capture

- **Status:** Accepted (Phase A shipped)
- **Related:** [ADR 0006](/decisions/0006_platform_view_capture_routing)

## Context

Flutter web targets use `WebVisualCapturePlatformAdapter` for permission metadata;
on a **macOS host**, `MacOsDesktopWindowScreenshotService` now also handles
`effectivePlatform: web` so `desktop_window` / `auto` can capture the Chrome window.

`HtmlElementView`, iframes, and WebGL/canvas content often appear as black holes in
`flutter_layer` captures — the same class of problem ADR 0006 solves via host pixels.

Agents running `flutter run -d chrome` on macOS need truthful screenshots without
assuming a broken renderer.

## Options investigated

| Approach                              | Mechanism                                                                                                                    | Pros                                                                                        | Cons                                                                                 |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| **A. macOS SCK on Chrome window**     | Extend `MacOsDesktopWindowScreenshotService` for `device: chrome` / `web` — candidates `Google Chrome`, `Chromium`, `Chrome` | Reuses ScreenCaptureKit, `focus`, and `desktop_capture_recovery`; same MCP modes as desktop | macOS host only; ambiguous with multiple Chrome windows; not Linux/Windows CI        |
| **B. Chrome DevTools Protocol (CDP)** | Parse browser debug port from `flutter run` output; MCP server calls `Page.captureScreenshot` over WebSocket                 | True tab pixels including DOM/canvas; works headless or headful                             | New port-discovery contract; security boundary (localhost); extra dependency surface |
| **C. Playwright / Puppeteer sidecar** | External process attaches to browser                                                                                         | Mature selectors and waits                                                                  | Heavy ops; second runtime; misaligned with single-binary CLI                         |
| **D. App / extension bridge**         | `mcp_toolkit` JS interop or DevTools extension screenshot                                                                    | Accurate for embedded DOM when app cooperates                                               | Not generic; requires per-app wiring                                                 |

## Decision

**Phase A (shipped):** Option A on macOS only — mirror iOS Simulator host capture.

- `supportsPlatform('web')` on macOS host.
- Window candidates: `google chrome`, `chromium`, `chrome` (owner + title match).
- Swift `activateChromeApp()` before capture when browser candidates are used.
- `auto` + `captureHints` from ADR 0006 apply (`HtmlElementView` strong signal).

**Phase B (deferred):** Option B (CDP) for Linux/Windows hosts and precise tab
selection when multiple Chrome windows run.

**Defer** C and D unless a concrete host-app driver appears.

## Consequences

**Easier**

- macOS agents on `flutter run -d chrome` can use `desktop_window` or `auto` for
  truthful browser pixels when Screen Recording is granted.

**Harder**

- Multiple Chrome windows: largest eligible window wins; agents may need to focus
  the correct tab manually or use `focus_window` before capture.
- Linux/Windows web hosts still rely on `flutter_layer` until Phase B.

## Notes

- Headless web (`chrome` without a visible window) is out of scope for Phase A;
  CDP remains the fit there.
- Flutter may expose `--web-browser-debugging-port`; CDP Phase B should parse
  machine/CLI discovery output rather than hard-code ports.
