# Built-in Tools

The Flutter MCP Toolkit comes with a set of pre-built tools and resources for common development tasks.

Notes:

- VM-dependent tools/resources resolve connection on demand. Startup does not force choosing a VM target.
- If VM service is unavailable, the tool returns an error payload.
- If multiple debug targets are discovered and no sticky/explicit target is available, the tool returns `connection_selection_required`.
- Error envelope and code semantics are defined in [Error Code Playbook](/core/error_code_playbook).

### Core Tools

- **`connect_debug_app`**: Explicitly select/connect a debug target.
- **`discover_debug_apps`**: List active debug targets with canonical websocket URIs.
- **`hot_reload_flutter`**: Performs a hot reload of the Flutter app.
- **`hot_restart_flutter`**: Performs a hot restart of the Flutter app.
- **`get_vm`**: Returns information about the Dart VM.
- **`get_extension_rpcs`**: Lists the available extension RPCs in the Flutter app.
- **`capture_ui_snapshot`**: Captures screenshots + view details + app errors in one bundle.
- **`inspect_widget_at_point`**: Resolves global (x,y) to the deepest widget/render node.

`get_active_ports` remains available in CLI for low-level workflows, but
`discover_debug_apps` is the preferred API for agent-driven target selection.

### Interaction Tools (drive the app like a user)

These tools let an AI agent interact with the running Flutter app the way a real user would. The interaction model mirrors Playwright MCP: get a compact semantic snapshot first, then use the returned `ref` strings to target specific widgets.

Every interaction tool uses a **two-tier dispatch**. Tier 1 invokes `SemanticsOwner.performAction` and directly runs the widget's registered handler (bypassing hit-testing, timing, and pointer-event synthesis). Tier 2 falls back to synthetic pointer events via `GestureBinding.instance.handlePointerEvent` when no matching semantic action is available. The response includes a `via` field (`semantic_action` | `pointer_events` | `editable_state`) so agents can tell which path was taken.

- **`semantic_snapshot`**: Returns a compact accessibility tree of interactive widgets only (buttons, fields, switches, sliders, etc.) with stable `ref` strings (`s_0`, `s_1`, ...) and a `snapshot_id` (monotonically incrementing int). Call this *before* any interaction tool. Refs are valid only against the most recent snapshot.
- **`tap_widget`**: Tap a widget by `ref`. Tier 1 via `SemanticsAction.tap` when the widget exposes `onTap`.
- **`enter_text`**: Enter text into a text field by `ref`. Tier 1 via `SemanticsAction.setText`; Tier 2 fallback focuses the field, then calls `EditableTextState.userUpdateTextEditingValue(..., SelectionChangedCause.keyboard)` so `TextInputFormatter`s and `onChanged` fire correctly.
- **`scroll`**: Scroll in `up` / `down` / `left` / `right` from a ref or screen center. Tier 1 via `SemanticsAction.scrollUp/Down/Left/Right` when `ref` is provided and the widget exposes it.
- **`long_press`**: Long-press a widget by `ref`. Tier 1 via `SemanticsAction.longPress`.
- **`swipe`**: Swipe in a direction. Always Tier 2 (pointer events with tight move cadence for fling velocity).
- **`drag`**: Drag from `fromRef` to `toRef`. Always Tier 2.

All interaction tools accept an optional `snapshotId` argument. If it's provided and doesn't match the current snapshot, the call returns `stale_snapshot` with `providedSnapshotId` and `currentSnapshotId` so the agent can re-snapshot without guessing.

### Tight Edit/Preview Cycle

- **`hot_reload_and_capture`**: Hot reload then immediately capture screenshot + semantic snapshot + errors in a single response. Designed for the AI iteration loop: edit code â†’ call this tool â†’ see what changed.

### Runtime Introspection

- **`evaluate_dart_expression`**: Evaluate a Dart expression in the running app isolate against the root library (e.g. `AgentState.instance.counter`). Returns `{result, kind, classRef}`. Lets agents inspect runtime state without requiring developers to pre-register custom tools.
- **`get_recent_logs`**: Retrieve recent `print()` / `debugPrint()` output from the running app (ring buffer, 200 entries).

### Golden Path (run against `flutter_test_app`)

The `flutter_test_app` showcase is designed for this exact flow. Launch it with `flutter run -d macos` in `flutter_test_app/`, read the VM URI from its output, then:

```bash
WS='ws://127.0.0.1:<port>/<token>/ws'

# 1. List interactive widgets
flutter_mcp_cli exec --name semantic_snapshot \
  --args "{\"connection\":{\"targetId\":\"$WS\"}}"

# 2. Increment the counter via semantic action
flutter_mcp_cli exec --name tap_widget \
  --args "{\"ref\":\"s_0\",\"connection\":{\"targetId\":\"$WS\"}}"

# 3. Fill the greeting field
flutter_mcp_cli exec --name enter_text \
  --args "{\"ref\":\"s_1\",\"text\":\"hello\",\"connection\":{\"targetId\":\"$WS\"}}"

# 4. Read runtime state directly
flutter_mcp_cli exec --name evaluate_dart_expression \
  --args "{\"expression\":\"AgentState.instance.counter\",\"connection\":{\"targetId\":\"$WS\"}}"

# 5. Scroll the list
flutter_mcp_cli exec --name scroll \
  --args "{\"direction\":\"down\",\"connection\":{\"targetId\":\"$WS\"}}"

# 6. Edit the "Hot reload marker" string in showcase_screen.dart, then:
flutter_mcp_cli exec --name hot_reload_and_capture \
  --args "{\"connection\":{\"targetId\":\"$WS\"}}"
```

### Known Limits

- **Refs are snapshot-scoped.** They're only valid against the most recent `semantic_snapshot`. Pass `snapshotId` to detect staleness.
- **Off-screen widgets aren't in the snapshot until scrolled into view.** `ListView.builder`-style scrollables lazy-render children, so the snapshot only contains nodes currently in the viewport + Flutter's cache extent (~250 logical px). For off-screen content: `scroll â†’ snapshot â†’ interact`.
- **Scroll with a `ref` is most reliable.** The semantic-action path (`SemanticsAction.scrollUp/Down/...`) is deterministic. Without a `ref`, the interaction layer falls back to `PointerScrollEvent` at screen centre â€” which can be swallowed by a full-screen overlay (e.g. a DevTools panel or the live-edit host) because hit-testing hits the topmost widget. If no-ref scroll seems stuck, grab a ref from the snapshot and pass it explicitly.
- **Widgets without semantics are invisible to the snapshot.** A raw `GestureDetector` without a `Semantics` wrapper won't appear. Reach it via `inspect_widget_at_point` + Tier 2 pointer events.
- **Custom text input routes bypass the fallback.** Text fields that don't use `EditableText` (e.g. custom platform views) can't be filled via `userUpdateTextEditingValue`; expose `SemanticsAction.setText` or set the value directly with `evaluate_dart_expression`.
- **Semantics tree is primed at `MCPToolkitBinding.initialize()` time** (debug mode only, negligible overhead). The first `semantic_snapshot` is synchronous; cold-path callers that skip the binding still work via an awaited frame.

CLI and MCP use the same shared command catalog/executor. You can verify quickly:

```bash

dart run mcp_server_dart/bin/flutter_mcp_cli.dart schema --name capture_ui_snapshot

```

### Connection Targeting (Per Request)

All VM-dependent tools accept an optional nested `connection` object in `arguments`:

```json
{
  "name": "get_vm",
  "arguments": {
    "connection": {
      "targetId": "ws://127.0.0.1:59490/<token>/ws"
    }
  }
}
```

Supported fields:

- `targetId` (full VM websocket URI)
- `mode` (`auto`, `manual`, `uri`)
- `host`
- `port`
- `uri`
- `forceReconnect`

Legacy `host:port` values for `targetId` are rejected.

Zero-mistake targeting:

1. Safest: use `connection.uri` with exact `app.debugPort.wsUri`.
2. If using `connection.targetId`, copy from `discover_debug_apps` or `availableTargets`.
3. If `target_not_found` occurs, retry with `connection.uri` and exact machine `wsUri`.

Flat top-level aliases like `arguments.port`, `arguments.host`, and `arguments.uri` are not supported.

The same nested object is used in CLI/daemon flows:

- one-shot `exec --args`
- daemon `command/execute` and `watch/start`
- snapshot step args (`commands[i].args.connection`)

### Ambiguity Handshake

When multiple targets exist and no sticky/explicit target resolves ambiguity, the server returns:

```json
{
  "code": "connection_selection_required",
  "details": {
    "reason": "multiple_targets",
    "availableTargets": [
      {
        "targetId": "ws://127.0.0.1:8181/<token>/ws",
        "host": "localhost",
        "port": 8181,
        "endpoint": "ws://127.0.0.1:8181/<token>/ws",
        "isSticky": false,
        "isCurrent": false
      }
    ],
    "suggestedAction": "retry_with_connection_target",
    "example": {
      "connection": {"targetId": "ws://127.0.0.1:8181/<token>/ws"}
    },
    "howToRetry": {
      "connection": {"targetId": "ws://127.0.0.1:8181/<token>/ws"}
    }
  }
}
```

### Hot Restart Tool Usage

The server now provides a dedicated hot restart tool for situations where hot reload isn't sufficient (for example after adding service extensions or when the app state is corrupted):

```json
{
  "name": "hot_restart_flutter",
  "arguments": {}
}
```

Expected result:

```json
{
  "message": "{\"report\":{\"type\":\"Success\",\"success\":true}}"
}
```

### Debug Dump Tools

These tools are disabled by default and can be enabled with the `--dumps` flag.

- **`debug_dump_layer_tree`**: Dumps the layer tree.
- **`debug_dump_semantics_tree`**: Dumps the semantics tree.
- **`debug_dump_render_tree`**: Dumps the render tree.
- **`debug_dump_focus_tree`**: Dumps the focus tree.

### Resources

- **`visual://localhost/app/errors/latest`**: Gets the latest errors from the app.
- **`visual://localhost/view/details`**: Gets details about the current view.
- **`visual://localhost/view/screenshots`**: Takes a screenshot of the app.

Resource reads also support connection targeting via URI query params:

- `targetId`
- `mode`
- `host`
- `port`
- `uri`
- `forceReconnect`

Example:

- `visual://localhost/view/details?targetId=ws%3A%2F%2F127.0.0.1%3A59490%2F%3Ctoken%3E%2Fws`
